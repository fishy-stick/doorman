# Deploy

Language: English | [简体中文](README.zh-CN.md)

This guide covers the production deployment options for Doorman:

- GHCR container image, recommended first
- GitHub Release binary archive, recommended for non-container deployments
- `systemd` service management
- Source builds, for development, debugging, or fallback use

Doorman is best deployed on a publicly reachable server outside the home network, such as a VPS, cloud instance, or another long-running public node. In production it runs as a Go binary with embedded frontend assets and depends on `config.yaml` plus a SQLite database file at runtime.

## Before Deployment

Before deploying, confirm that:

- The server can be reached from the target home network
- The external service URL is known and will be written to `server.public_url`
- You know whether Doorman will run behind a trusted reverse proxy
- The SQLite database directory will be persisted
- The firewall or security group allows the public service port, default `8080`

`server.public_url` affects the `curl` and `crontab` commands generated by the admin UI. Do not keep the default `http://your-server:8080` value for a real deployment.

## GHCR Container Deployment

Use the prebuilt image from GitHub Container Registry:

```bash
docker pull ghcr.io/fishy-stick/doorman:<version>
```

Replace `<version>` with a released tag, for example `v0.2-alpha`. Stable releases also publish `latest` and semantic aliases; prereleases should use their exact tag.

### Quick Trial

If you only want to confirm that the service starts, use the configuration built into the image:

```bash
docker run -d \
  --name doorman \
  -p 8080:8080 \
  -v doorman-data:/app/data \
  ghcr.io/fishy-stick/doorman:<version>
```

The default in-container configuration is equivalent to:

```yaml
server:
  port: 8080
  trust_proxy: true
  db: "/app/data/doorman.db"
  public_url: "http://your-server:8080"
```

Default runtime properties:

- The service listens on container port `8080`
- The database is stored at `/app/data/doorman.db`
- The container runs as `nobody:nogroup`
- The image declares `VOLUME /app/data`

As long as `/app/data` is persisted, the database survives container recreation.

### Production Configuration

For a real deployment, prepare your own `config.yaml`:

```yaml
server:
  port: 8080
  trust_proxy: true
  db: "/app/data/doorman.db"
  public_url: "https://your-domain.example"
```

Then mount the configuration file and data volume:

```bash
docker run -d \
  --name doorman \
  -p 8080:8080 \
  -v /path/to/config.yaml:/app/config.yaml:ro \
  -v doorman-data:/app/data \
  ghcr.io/fishy-stick/doorman:<version>
```

If you move `server.db` to another path, make sure the matching directory is mounted as well.

If you use a bind mount instead of a Docker volume, ensure the mounted directory is writable by `nobody:nogroup`; otherwise SQLite cannot create or update the database.

### First Startup and Checks

On first startup, the service generates an admin password and prints it to the logs:

```bash
docker logs doorman
```

Then open:

```text
http://<your-host>:8080/admin
```

Recommended initial checks:

1. Confirm `/admin` opens successfully
2. Sign in with the password from the logs
3. Create a network and trigger `/knock` once
4. Verify `/app/data/doorman.db` exists and history records are visible
5. Confirm the generated client command uses the expected `server.public_url`

## Release Binary Deployment

Use this path when you do not want to run containers. Release archives are published for Linux `amd64` and `arm64`, and each archive includes the embedded web UI.

Release asset names are:

- `doorman_${VERSION}_linux_amd64.tar.gz`
- `doorman_${VERSION}_linux_arm64.tar.gz`
- `SHA256SUMS`

### Download and Verify

Set the version and CPU architecture, then download the release archive and checksum file:

```bash
VERSION=v0.2-alpha
ARCH=amd64

curl -LO "https://github.com/fishy-stick/doorman/releases/download/${VERSION}/doorman_${VERSION}_linux_${ARCH}.tar.gz"
curl -LO "https://github.com/fishy-stick/doorman/releases/download/${VERSION}/SHA256SUMS"
sha256sum --ignore-missing -c SHA256SUMS
```

Use `ARCH=arm64` on ARM64 Linux hosts.

Extract the archive:

```bash
tar -xzf "doorman_${VERSION}_linux_${ARCH}.tar.gz"
cd "doorman_${VERSION}_linux_${ARCH}"
```

The extracted directory contains:

- `doorman`
- `config.example.yaml`

### Prepare Files and Configuration

A typical production layout is:

- `/opt/doorman/doorman`
- `/opt/doorman/config.yaml`
- `/var/lib/doorman/doorman.db`

Install the binary and example configuration:

```bash
sudo install -d /opt/doorman /var/lib/doorman
sudo install -m 0755 doorman /opt/doorman/doorman
sudo install -m 0644 config.example.yaml /opt/doorman/config.yaml
```

Edit `/opt/doorman/config.yaml`:

```yaml
server:
  port: 8080
  trust_proxy: true
  db: "/var/lib/doorman/doorman.db"
  public_url: "https://your-domain.example"
```

`server.public_url` is the external service URL used in generated `curl` and `crontab` commands. It may include a path prefix, such as `https://www.abc.com/prefix`; in that case, your reverse proxy must forward `/prefix/knock` to Doorman's `/knock` endpoint.

The program always reads `config.yaml` from the current working directory. If `server.db` uses a relative path such as `doorman.db`, that path is also resolved relative to the current working directory. Absolute paths are safer for production.

### Start Directly

For a direct shell run, start Doorman from the directory that contains `config.yaml`:

```bash
cd /opt/doorman
./doorman
```

Make sure the user running `./doorman` can write to the configured database directory, such as `/var/lib/doorman`.

After first startup, read the initial admin password from standard output logs.

## Running Under systemd

For long-running binary deployments, use a dedicated service user:

```bash
sudo useradd --system --home /opt/doorman --shell /usr/sbin/nologin doorman
sudo chown -R doorman:doorman /var/lib/doorman
```

Make sure the files are in this layout:

- `/opt/doorman/doorman`
- `/opt/doorman/config.yaml`
- `/var/lib/doorman/doorman.db`

Use this `systemd` unit:

```ini
[Unit]
Description=Doorman DDNS Service
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=doorman
Group=doorman
WorkingDirectory=/opt/doorman
ExecStart=/opt/doorman/doorman
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

Common commands:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now doorman
sudo systemctl status doorman
sudo journalctl -u doorman -f
```

On first startup, read the initial admin password from `journalctl`.

## Build From Source

Use source builds when you need to test local changes, debug the build, or target an environment not covered by the release artifacts. For normal production deployments, prefer the GHCR image or a GitHub Release binary.

The build machine needs Go `1.26+`, Node.js `22+`, and `pnpm`.

Build the frontend first, then compile the embedded binary:

```bash
cd web
pnpm install
pnpm run build:embed
cd ..

go build -tags embedweb -o doorman ./cmd/doorman/
```

You can also build a local image from the repository root:

```bash
docker build -t doorman:local .
```

Run source-built artifacts with the same `config.yaml`, data persistence, and `server.public_url` rules described above.

## Reverse Proxy and trust_proxy

If Doorman is behind Nginx, Caddy, or another reverse proxy that you control, you can keep:

```yaml
server:
  trust_proxy: true
```

In that mode, Doorman trusts these sources in order:

1. `X-Forwarded-For`
2. `X-Real-IP`
3. `RemoteAddr`

If the service is exposed directly to the public internet, or the upstream proxy chain is not fully trusted, use:

```yaml
server:
  trust_proxy: false
```

Otherwise, forged headers may affect public IP detection.

If `server.public_url` includes a path prefix, such as `https://www.abc.com/prefix`, configure the reverse proxy to forward `/prefix/knock` to Doorman's `/knock` endpoint.

## Upgrade and Change Notes

- When upgrading GHCR deployments, pull the new image tag and recreate the container while keeping the existing volume or bind mount
- When upgrading release binary deployments, replace `/opt/doorman/doorman` and keep the existing `config.yaml` and SQLite database
- Pin production deployments to a specific release tag when you want repeatable rollbacks; use `latest` only when tracking the newest stable release is intentional
- After changing `server.public_url`, newly generated client commands change, and old commands already copied to clients must be updated manually
- After regenerating a network token, all existing client commands stop working immediately
- Admin sessions are stored in memory, so a restart requires logging in again
- Before changing `trust_proxy`, confirm the real network topology and proxy chain
